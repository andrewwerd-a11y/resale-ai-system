# Resale AI System — Full Rundown

A local-first Python monorepo that automates eBay resale operations from photo intake through live listing. Runs on a single Windows machine. No cloud database, no Docker, no message queue required.

---

## What it does

1. **Intake** — scans SKU-named folders, normalizes photos to `NN.jpg` at 1600px, backs up originals, reserves SKUs atomically
2. **Vision extraction** — runs photos through local Ollama AI to extract title, description, condition, price estimate, and category-specific fields
3. **Triage** — routes items to single listing, lot, review queue, or reject based on confidence thresholds and rule triggers
4. **Review UI** — operator approves, edits, or rejects flagged items at `http://localhost:8000`
5. **Category Intelligence** — queries eBay Taxonomy API for required item specifics per category; flags missing fields before publish
6. **Enrichment** — optional Claude Sonnet pass that rewrites titles/descriptions; layers Browse API pricing on top
7. **Publishing** — full 3-step eBay API flow (inventory item → offer → publish) with Cloudinary photo hosting
8. **Sold sync** — pulls completed orders from eBay Fulfillment API, marks items sold, computes net profit

## End-to-end pipeline

```
intake/pending/CL-000099/01.jpg 02.jpg 03.jpg
          │
          ▼  FolderScanner.scan_pending()
          FolderManifest(detected_sku, image_paths, ...)
          │
          ▼  ImageNormalizer.normalize_folder()
          Renames to NN.jpg, resizes to 1600px, backs up originals to _original_backup/
          │
          ▼  SKURegistry.reserve(prefix) [new items only]
          Atomic increment via SKURepository
          │
          ▼  CategoryMapper.from_prefix("CL")
          {category_key, label, ebay_category_id}
          │
          ▼  build_extraction_prompt(category_key)
          Loads extraction_v1.txt + substitutes category-specific field schema
          │
          ▼  OllamaProvider.analyze(images[:3], prompt)
          POST /api/chat format=json → Result[JsonDict]
          │
          ▼  ResponseParser.parse(extracted, category_key)
          Type coercion, required field validation, review trigger detection
          (low confidence, high value, signed/first edition/luxury brand text scan)
          │
          ▼  PriceEstimator.apply(item)
          Derives list/min from estimated_price, enforces minimum_profit_margin
          │
          ▼  TriageRouter.route(item)
          SINGLE | LOT | REVIEW | REJECT
          │
          ▼  ItemRepository.upsert(item)
          INSERT or UPDATE — manual-override-protected fields preserved on update
          │
          ▼  If needs_review → ReviewRecord created
          Item visible in /review-queue or /inventory

─── operator action ────────────────────────────────────────

          ▼  Review UI: approve / edit / reject
          POST /api/review/{sku}/approve → status = export_ready
          PATCH /api/review/{sku}/edit → manual_override=True + status = export_ready

          ▼  [optional] enrichment pass
          scripts/enrich_all.py → ItemEnricher.enrich(item)
          Sends item JSON to Claude (text only, not images currently)
          PriceResearcher.research() layers Browse API avg over Claude's list_price

          ▼  [optional] category intelligence pass
          scripts/run_category_intelligence.py OR POST /api/items/{sku}/category-intelligence
          Suggestions API → leaf category_id
          Item Aspects API → required/recommended fields + allowed values
          CategorySpreadsheet caches template at data/category_intelligence/

          ▼  publish
          POST /api/ebay/publish/{sku}
          PhotoUploader → Cloudinary URLs (or local paths fallback)
          PUT /sell/inventory/v1/inventory_item/{sku}
          POST /sell/inventory/v1/offer
          POST /sell/inventory/v1/offer/{offerId}/publish
          Item.status = listed, listing_id + listing_url stored

─── post-listing automation ────────────────────────────────

          ▼  /api/ebay/sync-sold (cron-able)
          SoldSync.reconcile() → /sell/fulfillment/v1/order
          Match by SKU → status = sold, fees + net_profit computed

          ▼  /api/items/apply-stale-drops (cron-able)
          StaleChecker → drops list_price 10% on listings ≥60 days old (floor: minimum_price)

          ▼  cross-platform takedowns
          CrossPlatformSync.mark_sold() → ends listings on other platforms
          (currently only logs warnings — no other platform APIs implemented yet)
```

## Data model

### Six tables

| Table | Model | PK | Purpose |
|---|---|---|---|
| `items` | `ItemRecord` | `internal_id` (uuid), `sku` unique | Central item record — ~70 columns |
| `review_cases` | `ReviewRecord` | `review_case_id` (uuid) | One row per review event with trigger reasons + confidence |
| `sale_records` | `SaleRecord` | `id` (uuid) | One row per sale with computed gross/net profit + margins |
| `sku_registry` | `SKURecord` | `prefix` | Tracks last reserved number per prefix for atomic SKU generation |
| `sourcing_batches` | `SourcingBatch` | `batch_id` (uuid) | Bulk purchase events — drives cost-per-item assignment |
| `batches` | `BatchRecord` | `batch_id` (uuid) | One row per worker run — tracks processed/failed counts |

### Item lifecycle

```
pending_intake
   │  (worker runs vision pipeline)
   ▼
analyzed ──┬─→ needs_review ──┬─→ approved ──→ export_ready ──┬─→ exported (CSV path)
           │                  │                                │
           │                  │                                └─→ listed ──→ sold ──→ archived
           │                  └─→ rejected
           └─→ rejected (auto-reject from triage)

Lot wrappers:
   approved item → LotBuilder.create_lot()
   → wrapper gets item_mode="lot", members get status="lot_member"
```

Status transitions are not enforced by code — any handler can set any status. The conventions above are the contract.

### ItemRecord field groups (~70 columns)

| Group | Fields |
|---|---|
| Identity | `internal_id`, `sku`, `status`, `item_mode`, `batch_id` |
| Photos | `photo_folder`, `image_paths` (pipe-separated string) |
| Category | `category_key`, `category_label`, `ebay_category_id`, `ebay_category_name`, `category_template_fetched`, `missing_required_fields`, `missing_recommended_fields`, `publish_ready` |
| Titles | `title_raw`, `title_final`, `description_final` |
| Brand | `brand`, `brand_normalized` |
| Item specifics | Clothing (size, color, material…), book (author, publisher, isbn…), collectible/toy (mpn, upc, character, franchise…) |
| Condition | `condition_label`, `condition_id`, `condition_notes`, `defects` |
| Measurements | JSON dict |
| Lot | `bundle_candidate`, `lot_group_id`, `lot_reason` |
| Pricing | `cost`, `estimated_price`, `list_price`, `minimum_price`, `shipping_*`, `storage_location` |
| Listing | `platform`, `listing_id`, `listing_url`, `date_listed`, `days_listed`, `date_sold`, `sold_price`, `fees`, `shipping_cost`, `net_profit`, `profit_margin` |
| AI / review | `confidence_score`, `needs_review`, `review_reasons`, `manual_override`, `notes` |
| Enrichment | `enrichment_done`, `enrichment_notes` |
| Sourcing | `sourcing_location`, `sourcing_date`, `sourcing_batch` |
| Catch-all | `item_specifics` (JSON dict — category-specific aspects beyond typed columns) |
| Timestamps | `created_at`, `updated_at` |

### JSON serialization in SQLite

Fields typed as `list`/`dict` on the Pydantic `Item` entity are stored as JSON strings in SQLite. `image_paths` is pipe-separated. Conversion happens in `_to_record()` / `_from_record()` in `packages/data/src/repositories/item_repo.py`. If you add a list/dict field, register it in the serializer's known-JSON-field set or it will coerce to comma-joined string and break deserialization.

JSON string fields: `features`, `defects`, `review_reasons`, `measurements`, `item_specifics`, `missing_required_fields`, `missing_recommended_fields`

## Invariants

These are properties the code maintains. Violating any of them is a bug.

1. Once `manual_override=True`, never overwrite `cost`, `list_price`, `minimum_price`, `notes`, `storage_location` from AI/automation. (Enforced in `ItemRepository.upsert()`.)
2. Once `cost_manual=True`, never reset to False. (Same.)
3. Once `enrichment_done=True`, never reset to False. (Same.)
4. `internal_id` and `created_at` are immutable. Updates filter them out.
5. SKU prefix is authoritative for category. No code path overrides category from AI extraction.
6. The same SKU never has two rows. Upsert keys on `sku` (unique index + upsert logic).
7. The SKU registry never goes backwards. `reserve_next` only increments; `preserve_existing` raises the floor, never lowers it.
8. The bulk-upload CSV never duplicates lot members. `EbayCSVWriter.write()` filters out items with `status="lot_member"`.
9. Vision model receives at most 3 images. Hardcoded in `OllamaProvider.analyze` for context-window stability.
10. Image filenames are normalized to `NN.jpg`. Folder scanner assumes this; image normalizer enforces it (with backup of originals).

## Design principles

### 1. Local-first
Runs entirely on one machine. SQLite, local filesystem, local Ollama. Internet needed only for: initial Ollama model pull, Cloudinary uploads (optional), eBay API calls, Claude API enrichment (optional).

### 2. Prefix is authoritative for category
The two-letter SKU prefix (`BK`, `CL`, `CO`, `SH`, `TO`) determines category — not AI classification, not the eBay Suggestions API. AI fills fields *within* a category but never overrides which category an item belongs to. To add a new category: edit `config/sku_prefixes.json` + `config/categories.json`. No code changes.

### 3. Manual override wins, always
Once a human edits a field via the review UI (or any `PATCH /api/items/{sku}`), `manual_override=True` is set. The AI pipeline will not write to protected fields again, even on reprocessing with better photos or a smarter model.

### 4. Idempotent upserts
Every write to the items table goes through `ItemRepository.upsert()`. Re-running the worker on the same SKU updates fields in place without creating duplicates. Re-running the backfill on the same source folder is safe.

### 5. Result[T] everywhere
Cross-module calls return `Result[T]` instead of raising. This forces callers to acknowledge failure paths explicitly.

```python
@dataclass
class Result(Generic[T]):
    ok: bool
    value: T | None = None
    error: str | None = None           # human-readable
    error_code: str | None = None      # machine-readable ("NOT_CONFIGURED", "API_ERROR")
    details: dict = field(default_factory=dict)  # diagnostic data (eBay response body, etc.)
```

Exceptions are reserved for programming bugs and unrecoverable startup failures.

### 6. Configuration over code
Behaviour-shaping decisions live in `config/*.json`. Triage thresholds, review trigger keywords, luxury brand lists, eBay column mappings, model profiles — all editable without a code change. Config accessors are `@lru_cache(1)`. The `PATCH /api/settings/rules` endpoint manually clears `get_rules.cache_clear()` so changes take effect mid-process.

### 7. Two layers of "missing" detection
A field can be missing from our schema (AI didn't extract it — caught by `ResponseParser` against `categories.json`) or missing from eBay's category requirements (caught by Category Intelligence against the Taxonomy API). Both paths flow into `review_reasons` but are driven by independent rules.

## External integrations

### Ollama (local vision)

- **Module:** `packages/vision/src/ollama_provider.py`
- **Endpoint:** `POST {ollama_base_url}/api/chat`
- **Params:** `format=json`, `temperature=0.1`, `num_ctx=8192`, `repeat_penalty=1.15`
- **Image limit:** First 3 images per item (front/back/tag)
- **Retries:** 3 attempts, exponential backoff via `tenacity`
- **Failure modes:** Connection error, HTTP error, JSON parse failure (handled by `_extract_json` brace-balancing fallback)
- **Dry-run:** `Settings.dry_run=True` returns stub response without calling Ollama

### Anthropic / Claude (enrichment)

- **Module:** `packages/enrichment/src/enricher.py`
- **Model:** `claude-sonnet-4-20250514` default
- **Mode:** Text-only (item JSON in, enriched JSON out — does not currently send images)
- **Cost:** ~$0.02/item; estimated cost returned in `Result.details["estimated_cost"]`
- **Price layering:** After Claude responds, `PriceResearcher.research()` is called; Browse API average overrides Claude's `list_price` if available
- **Protected fields never written by enrichment:** `sku`, `status`, `batch_id`, `photo_folder`, `image_paths`, `category_key`, `category_label`, `ebay_category_id`, `internal_id`, `enrichment_done`, `cost_manual`, `created_at`, `updated_at`

### Cloudinary (photo hosting)

- **Module:** `packages/ebay/src/photo_uploader.py`
- **Upload folder:** `resale/`
- **Fallback:** Local file paths if Cloudinary not configured (works for sandbox, breaks production listings)

### eBay (multiple APIs)

All eBay code lives in `packages/ebay/src/`. `EbayAuth` is the single source of truth for tokens.

| API | Module | Auth | When called |
|---|---|---|---|
| OAuth 2.0 | `auth.py` | basic auth (app_id + cert_id) | `/api/ebay/oauth/*` flow + auto-refresh |
| Inventory | `inventory_client.py` | OAuth user token | Each publish |
| Account | `inventory_client.py` | OAuth user token | First publish per process (cached) |
| Browse | `pricing/price_researcher.py` | App token (client_credentials) | Inside enrichment pass |
| Taxonomy | `category_intelligence.py` | App token (falls back to user) | Category suggestions + item aspects |
| Fulfillment | `sold_sync.py` | OAuth user token | `/api/ebay/sync-sold` |

**Token lifecycle:**
1. Operator clicks Connect eBay in UI → `/api/ebay/oauth/start` → 302 to eBay
2. eBay calls back to `/api/ebay/oauth/callback?code=…`
3. `EbayAuth.exchange_code_for_tokens()` saves `data/ebay_tokens.json`
4. Subsequent calls check expiry; if within 5 min, silent refresh via `_refresh_access_token`
5. If file is missing/expired-without-refresh, falls back to `EBAY_*_USER_TOKEN` from `.env`

**Production vs sandbox:** Controlled by `EBAY_ENVIRONMENT`. The Taxonomy host is always production (`https://api.ebay.com`) regardless — sandbox doesn't have meaningful catalog data.

**Seller policies:** First active policy of each type (fulfillment, payment, return) is selected and cached. No UI to choose among multiple policies.

### SMTP (notifications)

- **Module:** `packages/notifications/src/notifier.py`
- **Activation:** `notifications_enabled=true` in `.env`
- **Failure mode:** Logs warning, never raises — notification failure must never break the pipeline

### Watchdog (file watcher)

- **Module:** `packages/capture/src/file_watcher.py`
- **Activation:** `POST /api/capture/watcher/start`
- **Current behavior:** Observes and logs new image files — does not trigger the worker yet

## Concurrency model

There isn't one, deliberately.

- FastAPI server is a single uvicorn process
- Worker is a single subprocess that reads + writes serially
- SQLite uses `check_same_thread=False` but is effectively serialized
- No row-level locking, no transaction isolation handling, no optimistic concurrency

Running multiple workers in parallel risks SKU registry collisions, photo upload duplication, and last-write-wins on item updates. "Run one worker at a time" is part of the contract. Phase 7 (live capture studio) will need to revisit this.

## Extension points (no code changes required)

### Add a new category
1. Edit `config/sku_prefixes.json` — add `{label, category_key, ebay_category_id, lot_eligible, active}`
2. Edit `config/categories.json` — add profile with `required_fields`, `optional_fields`, `allowed_values`, `review_triggers`, `title_template`, `description_template`, `lot_grouping_keys`
3. (Optional) Add review triggers in `config/rules.json` under `review_triggers.<category_key>`
4. Restart server (config is `lru_cache`'d)

### Add a new vision provider
1. Create `packages/vision/src/<provider>_provider.py` implementing `VisionProvider` ABC
2. Add a profile entry in `config/model_profiles.json`
3. Wire in via constructor param in worker or `/api/items/{sku}/analyze` route

### Add a new HTTP endpoint
1. Pick the right file in `apps/api/src/routes/` (or create one)
2. Add a thin handler — delegate logic to a package
3. Register the router in `apps/api/src/main.py` if it's a new file
4. Run `make docs` to regenerate `docs/openapi.json`
